## Phase Goal

**As a** maintainer of an Nx workspace whose CI runs on GitHub Actions, **I want to** have my
Nx remote cache read and write GitHub's own Actions cache automatically -- writable only on
trusted triggers, read-only everywhere else -- **so that** CI tasks get real cache hits with
nothing extra to host and no untrusted trigger can ever poison the cache.

> Slice for this plan: write safety under concurrency. Nx fires many cache writes at once. Two
> writes for the SAME hash would otherwise race on one shared temp archive file, and a truncated
> archive uploaded to the shared cache is a wrong artifact served to every later reader. After
> this plan, same-hash writes queue and different-hash writes still run in parallel.

<objective>
Build the per-hash serialization primitive, test-first: `withHashLock(hash, fn)` chains
same-hash operations through an in-process `Map<string, Promise<unknown>>`, runs different
hashes concurrently, evicts each map entry once its tail settles, and never lets a rejected
operation wedge the queue.

Purpose: TEST-02 -- the concurrency rigor the previous proof-of-concept's version lacked. The
four properties are one requirement each; the failure mode of getting any of them wrong is
silent (a wedged queue looks like a hang, an un-evicted map looks like a slow leak).

Output: `src/lib/with-hash-lock.ts` + `src/lib/with-hash-lock.spec.ts`. This plan does NOT wire
the lock into the write path -- that composition happens in Plan 02-05, at the same single point
where SIGTERM in-flight tracking attaches.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| concurrent in-process writers -> one shared temp archive file | Two same-hash writes target the identical temp path; without serialization one truncates the other mid-upload. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-2-06 | Tampering (data corruption) | concurrent same-hash writes to the shared archive path | high | mitigate | `withHashLock` serializes every operation for a given hash through a chained promise, so only one writer ever holds the shared temp path. A truncated or interleaved archive uploaded under a valid hash key would be served as a correct artifact to every later reader -- corruption here is indistinguishable from a legitimate hit, which is why serialization is a correctness control and not an optimisation. |
| T-2-07 | Denial of Service | wedged lock queue / unbounded map growth | medium | mitigate | The no-wedge property (chaining through both settle branches) and the identity-checked eviction are each asserted by a dedicated test. A wedged queue would hang every subsequent write for that hash; an un-evicted map would leak one entry per distinct hash for the process lifetime. |
| T-2-08 | Repudiation | swallowed operation failure | medium | mitigate | The map holds a non-rejecting tail but the caller receives the real result promise, so a failed write still rejects to its caller and reaches the server's fail-closed write path rather than being silently absorbed by the lock. Asserted by the no-wedge test. |
| T-2-SC | Tampering | npm/pip/cargo installs | high | accept | No package-manager install occurs in this plan; Node standard library only. Accepted -- the install vector does not occur here. |
</threat_model>
