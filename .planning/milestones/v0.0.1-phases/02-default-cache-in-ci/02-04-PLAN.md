## Phase Goal

**As a** maintainer of an Nx workspace whose CI runs on GitHub Actions, **I want to** have my
Nx remote cache read and write GitHub's own Actions cache automatically -- writable only on
trusted triggers, read-only everywhere else -- **so that** CI tasks get real cache hits with
nothing extra to host and no untrusted trigger can ever poison the cache.

> Slice for this plan: real storage. Every prior plan in this project has stored cache entries in
> a `Map` that dies with the process. After this plan a `CacheBackend` exists whose `put` really
> uploads to GitHub's cache service and whose `get` really downloads from it.

<objective>
Build the first real storage backend, test-first: `createActionsCacheBackend()` satisfies the
existing `CacheBackend` port by mapping `get` to `restoreCache` and `put` to `saveCache`, with
every path string resolved through one exported `cacheArchivePath(hash)` helper.

Purpose: ROBUST-03 and ROADMAP SC5 -- this is the module the CI dogfood exercises for a real hit.
The archive-path single-source rule is the phase's highest-severity risk: `@actions/cache`
version-hashes the LITERAL path string, so a cosmetic change to that string silently changes the
version and every restore MISSes, with no error anywhere.

Output: `src/lib/cache-archive-path.ts` (+ a pinned exact-name spec) and
`src/backend/actions-cache-backend.ts` (+ a spec against a mocked toolkit module).
</objective>

## Artifacts this phase produces

See `02-01-PLAN.md` "## Artifacts this phase produces" for the full Phase 2 symbol list. This
plan creates `cacheArchivePath`, `createActionsCacheBackend` and `cacheKeyFor`.

**Resolved planning decision (do not re-open):** `cacheArchivePath` ships as its OWN module,
not as a private function inside the backend. 02-PATTERNS.md makes the separate module
conditional on writing a pinned-exact-string test, and this plan writes exactly that test. The
reason the pinned test is worth a module: asserting the path only through the toolkit mock's call
arguments proves that save and restore agree with EACH OTHER, which stays true after a cosmetic
rename -- precisely the change that breaks every restore. Pinning the produced file name in a
spec is the only assertion that fails on that change. The in-repo precedent is
`MAX_CACHE_BODY_BYTES`, an exported load-bearing constant whose exact value its own spec pins.

**Resolved planning decision (do not re-open):** no `src/backend/read-only-backend.ts` is
created, in this plan or any other. See `02-05-PLAN.md` for the rationale.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| this process -> GitHub's cache service | Bytes leave the runner and become readable by every later job in the repository; the toolkit is the only sanctioned crossing. |
| archive path string -> toolkit version hash | The literal path is an input to the key derivation, so the string itself is a correctness-critical value, not a formatting detail. |
| local temp filesystem -> uploaded artifact | Cache bytes transit a world-readable-per-user temp file before upload and after download. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-2-09 | Tampering | `cacheArchivePath` literal string (silent-MISS class) | high | mitigate | Exactly one exported helper produces the string (repository-wide count asserted), both toolkit call sites resolve through it, the produced file name is pinned by a spec that spells the literal out rather than reconstructing it, and a comment-lock names the pitfall and forbids cosmetic edits without an end-to-end CI restore. The end-to-end confirmation is the Plan 06 canary. |
| T-2-10 | Repudiation / integrity | ambiguous `saveCache` sentinel and reserve-conflict error | medium | mitigate | Both are read as a benign no-op yielding `stored`, per the locked decision, and the comment states that the safety of that reading comes from the upstream write gate rather than from the backend. Crucially, only these two specific outcomes are absorbed: every other rejection propagates so the server's fail-closed write path surfaces a 500 instead of a silent success. Asserted by a dedicated propagation test. |
| T-2-11 | Information disclosure / resource exhaustion | temp archive file left on disk | low | mitigate | `put` removes the temp file in a `finally` block, so cleanup runs on the success path and on the propagating-error path alike; asserted on both paths. Cache bytes are not left behind on a shared or reused runner. |
| T-2-12 | Tampering | hand-rolled protocol drift | medium | mitigate | The backend calls only the official toolkit's `restoreCache` / `saveCache`. No direct call to the cache service endpoint is made, so protocol, compression selection, chunking and version-hash derivation stay owned by the pinned dependency. |
| T-2-SC | Tampering | npm/pip/cargo installs | high | accept | No package-manager install occurs in this plan; it consumes the dependency installed under a blocking human legitimacy checkpoint in Plan 02-01. Accepted -- the install vector does not occur here. |
