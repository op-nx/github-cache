## Phase Goal

**As a** maintainer of an Nx workspace whose CI runs on GitHub Actions, **I want to** have my
Nx remote cache read and write GitHub's own Actions cache automatically -- writable only on
trusted triggers, read-only everywhere else -- **so that** CI tasks get real cache hits with
nothing extra to host and no untrusted trigger can ever poison the cache.

> Slice for this plan: the whole capability becomes real. After this plan, starting `serve` on a
> trusted CI trigger gives you a loopback Nx cache server backed by GitHub's Actions cache;
> starting the same command anywhere else gives you a server that answers reads and refuses every
> write with a 403 -- with nothing the operator can pass to change that.

<objective>
Wire the pieces into the composition root, test-first. `selectBackend(env)` picks exactly one
backend per process from runtime context alone; `serve()` calls it in place of the hard-wired
writable memory backend, decorates the write path with the per-hash lock plus in-flight
tracking, and drains those in-flight writes on SIGTERM within a bounded grace period.

Purpose: TEST-01 (the selection case matrix), TRUST-05 (the load-bearing no-caller-facing-mode
property, proved both structurally and behaviorally), and ROBUST-04 (SIGTERM drains in-flight
writes so a background-step teardown does not lose the last write). This also completes the
TEST-02 wiring by attaching `withHashLock` to the real write path.

Output: `src/lib/select-backend.ts` + its spec, and modified `src/serve.ts` + `src/serve.spec.ts`.
</objective>

See `02-01-PLAN.md` "## Artifacts this phase produces" for the full Phase 2 symbol list. This
plan creates `selectBackend` and `resolveGitHubToken`, and adds `shutdown` to `RunningServer`.

**Resolved planning decision (do not re-open): no `src/backend/read-only-backend.ts` is
created.** `createReadOnlyMemoryBackend()` already IS the 403 seam, is already covered by
`memory-backend.spec.ts`, and is named in 02-CONTEXT.md as the reusable analog. A new module
would be a rename of working, tested code that Phase 3 replaces wholesale with the real
cross-context reader -- and an unused re-export would additionally trip the repository's
dead-code gate. `selectBackend` imports the existing factory directly, with a comment recording
that this is a deliberate Phase 3 placeholder.

**Resolved planning decision (do not re-open): `resolveGitHubToken` is a real precondition, not
a decorative one.** Nothing in Phase 2 sends the token anywhere -- the Actions-cache primitive
authenticates with its own runtime token -- but TEST-01 requires the fallthrough to be
implemented and specified, and Phase 3's authenticated private-repo read consumes it. Treating an
unresolvable token as "not a trusted write context" (degrade to read-only) rather than as a throw
keeps a merely-unwired workflow from breaking the build, while a MALFORMED `GITHUB_REPOSITORY`
does throw, because a corrupted repository identity in a trusted context is a misconfiguration
that must not be papered over silently. The Plan 06 dogfood job therefore passes the workflow
token through the job environment.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| runtime context -> write capability | The composition root decides once, at process start, whether this process can write to a shared cache. Everything after that point holds a backend that either can or cannot write. |
| operator / caller -> the selection decision | Any surface that lets a caller influence the choice would let a misconfiguration or a hostile workflow edit obtain write access without changing the trigger. |
| runner teardown signal -> in-flight write | Between SIGTERM and the runner's kill there is a short window in which a partially completed write can be lost or, worse, block cleanup. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-2-13 | Elevation of privilege | a caller-facing mode surface on the selection path | critical | mitigate | `selectBackend` takes only the env bag; there is no options object, no second argument and no environment variable that requests the writable backend. Proved twice: structurally (the function declares one defaulted parameter, asserted on `selectBackend.length`) and behaviorally (an untrusted env bag carrying override-shaped extra keys still yields a backend whose `put` is refused). The behavioral half exists because this repository has already shipped a tautological security test; an identity-only assertion would not have caught a smuggled flag. |
| T-2-14 | Spoofing / Tampering | malformed `GITHUB_REPOSITORY` in trusted context | high | mitigate | Validated against an `owner/name` shape and rejected with a throw at construction, following the existing fail-closed construction-guard precedent. A corrupted repository identity in a write-trusted context is a misconfiguration that must fail loudly rather than resolve into some other repository's cache namespace. |
| T-2-15 | Denial of service | unbounded SIGTERM drain | high | mitigate | The drain awaits in-flight writes only up to a bounded grace period backed by an unref'd timer, so a hung write yields to the runner's kill instead of blocking the implicit wait before job cleanup. Asserted by a test in which the gated write never settles and shutdown still resolves. |
| T-2-16 | Tampering (data loss) | writes lost at teardown | medium | mitigate | Every write is tracked in an in-flight set at the same wrapper that applies the per-hash lock, and shutdown awaits that set before closing out, so a write already accepted by the server is not silently dropped when the background step is cancelled. Asserted by the in-flight drain test. |
| T-2-17 | Information disclosure | bearer token handling in the composition root | medium | accept | The token fallthrough and CSPRNG minting are unchanged Phase 1 behavior and remain covered by the existing specs; this plan does not alter how the token is produced or compared. The masking of that token in CI logs is Plan 06's responsibility. Accepted here -- no new exposure is introduced. |
| T-2-SC | Tampering | npm/pip/cargo installs | high | accept | No package-manager install occurs in this plan. Accepted -- the install vector does not occur here. |
