## Phase Goal

**As a** developer working on any OS, **I want to** read this repo's CI-produced cache locally
through the GitHub Releases reader using my existing GitHub auth, **so that** I get cache hits
without ever being served a wrong-OS artifact.

> Source: ROADMAP.md Phase 3 `**Goal:**`, re-rendered into the three user-story slots. The
> capability and outcome are carried verbatim; nothing is invented.

<objective>
Deliver the OS-correct half of the read slice: an asset-name scheme that folds the running
platform into every entry, and a `CacheBackend` reader that derives every name through that one
helper, answers `'forbidden'` to every write, and degrades every fault to a MISS.

After this plan the reader is complete and provably never serves a wrong-OS artifact against an
injected client. Plan 03 attaches the real GitHub client and wires it into `selectBackend`.

Purpose: CORR-01 (never a wrong result) and TEST-05 (the cross-OS regression guards) are the
correctness core of the phase. They are settled first because Phase 4's publisher inherits this
exact key scheme -- a drift between the two derivations is a silent cross-OS MISS.

Output: `release-asset-name.ts` (+spec), `releases-backend.ts` (+spec).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| injected client -> backend `get` | An arbitrary `ReleaseReadClient` implementation (real or fake) may reject or return arbitrary bytes; the port must not let a rejection escape. |
| backend -> stderr | Warning text crosses into the developer's build log, which is frequently pasted into issues and CI transcripts. |
| task hash -> asset name | A `{hash}` value reaches a name that Plan 03 interpolates into a URL path. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Tampering | `createReleasesReadBackend.get` | high | mitigate | OS-namespace every entry through `releaseAssetName` and prove it with the never-wrong-OS NEGATIVE assertion; a wrong-OS artifact served as a valid hit is a wrong RESULT, the project's core-value violation. |
| T-03-02 | Denial of Service | `createReleasesReadBackend.get` | medium | mitigate | One `try/catch` at the port boundary converts every fault -- including an injected client that throws -- to `{ kind: 'miss' }`, so a read fault can never break a build (SRV-05, D-11). |
| T-03-03 | Information Disclosure | one-time stderr warning | high | mitigate | Warning text is a fixed ASCII sentence: the caught error is not interpolated, subprocess output is never echoed, and no token value is reachable from this module. |
| T-03-04 | Elevation of Privilege | `createReleasesReadBackend.put` | high | mitigate | `put` declares no parameters and has exactly one return path, `'forbidden'`; read-only is the absence of a write path, not a disabled feature (D-02, TRUST-05). Asserted for every input shape. |
| T-03-05 | Spoofing | `ReleaseReadClient` seam | medium | accept | The seam is a constructor argument on an unexported-from-barrel factory in the same package; `selectBackend` is the only production construction site and always passes the real client. No env value or caller argument can swap it, so the residual risk is in-package code review, not runtime. |
| T-03-06 | Denial of Service | warning volume | low | mitigate | Module-level once-per-process flag, and silence on the ordinary absent-asset path, so a cold cache does not spam every build line. |
</threat_model>
