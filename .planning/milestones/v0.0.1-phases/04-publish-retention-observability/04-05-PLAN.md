## Phase Goal

**As a** repo maintainer running Nx builds in CI, **I want to** mirror CI-produced cache entries to GitHub Releases on default-branch pushes and prune them safely by age, with loud failures and graceful cap handling, **so that** cross-context readers get correct cross-OS hits and the cache never breaks a build or deletes live data.

This slice makes pruning real: expired assets actually get deleted daily, safely and loudly.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cleanup credential -> Release delete | The delete path must use only a job-scoped, least-privilege same-repo token; a foreign/over-scoped credential is an elevation risk |
| npm install -> build tree | A new dependency is supply-chain surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-15 | Elevation of Privilege | cleanup credential | high | mitigate | Same-repo `contents:write` `GITHUB_TOKEN` only, no PAT, no `actions:read`, no `delete:packages` (Releases needs none) — RETAIN-03/C11; token by process inheritance |
| T-04-16 | Denial of Service / data loss | concurrent cleanup writers | high | mitigate | Single scheduled job + `concurrency` group with `cancel-in-progress: false` (queue, never cancel mid-delete) — D-09; no OS matrix, no ci.yml embedding |
| T-04-SC | Tampering | @octokit/rest install | high | mitigate | Exact-pin 22.0.1 (verified OK in the Legitimacy Audit: official octokit org, no postinstall) + pinned-deps.spec.ts range guard. Non-blocking: package verdict is OK, so no human legitimacy checkpoint |
