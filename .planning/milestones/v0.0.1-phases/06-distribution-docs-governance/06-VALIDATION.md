## Audit Trail (retroactive, post-execution — gsd-nyquist-auditor, 2026-07-21)

**Method:** Read all 5 phase-6 PLAN/SUMMARY pairs, `06-VERIFICATION.md`, `.planning/REQUIREMENTS.md`,
and the 5 shipped guard specs (`public-surface.spec.ts`, `docs-trust.spec.ts`,
`docs-adoption.spec.ts`, `governance-email.spec.ts`, `pinned-deps.spec.ts`) directly — not
accepting the SUMMARY/VERIFICATION coverage claims at face value. Cross-checked each spec's actual
assertions against the real production artifacts it claims to guard (`start-cache-server/action.yml`,
root `SECURITY.md`/`LICENSE`, `README.md`, `docs/*.md`). Ran `npx nx test github-cache` fresh (not
relying on a cached pass) both before and after adding new coverage.

**Findings, per requirement:**

- **DOCS-05** — CONFIRMED covered, strongly. `public-surface.spec.ts` is a genuine explicit-
  assertion-list guard (not a snapshot), and the RED→GREEN drift-detection property is proven with
  real git commits (`098598a` failing on a deliberately-wrong `serve` export, `9ff8ba8` fixed to
  451/451). No gap.
- **DOCS-02** — CONFIRMED covered. `docs-adoption.spec.ts` checks all 7 env knobs by name, the
  `MAX_CACHE_BODY_BYTES`+"fixed" pairing, the "10 GB" note, and the no-local-read phrase, all
  against the real `docs/configuration.md` on disk. No gap.
- **DOCS-05 mirrors + GOV-03/DOCS-03** — `docs-trust.spec.ts`'s single-source drift design
  (importing the REAL `TRUSTED_EVENTS`/`HOST_GATED_EVENTS`/`SYNC_EVENTS` rather than hardcoding
  event strings) is a genuinely strong, non-vacuous guard: verified by reading `lib/trust.ts` and
  `lib/sync-gate.ts` directly that the imported arrays are the actual production allowlists, not a
  spec-local copy that could drift from them independently. No gap in the automatable portion; the
  security-literate semantic read is correctly Manual-Only (already flagged as `human_judgment:
  true` in 06-05-SUMMARY.md).
- **DOCS-01/DOCS-04** — CONFIRMED covered for the structural/presence facts. Verified the negative
  assertion in DOCS-04's guard (`does not include 'operation:'`/`'matrix:'`) is NON-VACUOUS by
  confirming both tokens actually appear in `.github/workflows/ci.yml` (`operation:` present in the
  publish/cleanup jobs; `matrix:` present in the cross-OS jobs) — so the guard would genuinely catch
  a regression where the minimal example grew a matrix or an `operation:` field. The "visibly
  smaller" size/prose judgment is correctly Manual-Only.
- **DOCS-06 — GAP CONFIRMED and FILLED.** The plan's own prohibition states the consumer action
  "MUST be a JS action ... NOT composite" because a composite action cannot declare `background:`
  internally (research Pitfall 5) — this is the single most safety-critical structural invariant
  DOCS-06 depends on, since regressing `runs.using` to `composite` would silently break every
  consumer's background-step CI with no other test catching it. Searched every phase-6 spec file
  (`public-surface.spec.ts`, `docs-adoption.spec.ts`, `docs-trust.spec.ts`) for any assertion on
  `start-cache-server/action.yml`'s `runs.using` or `runs.main` fields: none exists.
  `public-surface.spec.ts` parses ONLY the `inputs:` block of that file; the sibling
  `ppe/ppe-action.spec.ts` asserts `using: composite` but for the UNRELATED `ppe/action.yml`
  (advisory PPE gate), not `start-cache-server/action.yml`. Generated
  `packages/github-cache/src/consumer-action-runtime.spec.ts` (3 tests): asserts `runs.using` is
  `node24`, asserts `runs.using` is NOT `composite`, and asserts `runs.main` resolves to exactly
  `index.js` (never a `dist/` path). **Adversarial RED proof (test-only mutation, never touching the
  tracked `action.yml`):** temporarily flipped the test's own "not composite" assertion to require a
  composite match and re-ran with `--skip-nx-cache` — it failed exactly as expected against the real
  file (proving the assertion direction is meaningful, not tautological); restored the correct
  assertion and confirmed green. During initial authoring the third assertion (`not.toContain('dist/')`)
  was too broad and failed against the file's own explanatory COMMENT text (which legitimately
  mentions "gitignored dist/" in prose) — fixed by scoping the assertion to the `main:` field's
  parsed value specifically, not a whole-file substring search (debug iteration 1/3, root cause: test
  expectation too broad, not an implementation defect).
- **GOV-01 — GAP CONFIRMED and FILLED.** `governance-email.spec.ts` (the only ongoing spec touching
  `SECURITY.md`) scans it EXCLUSIVELY for email-shaped tokens; it asserts nothing about the
  disclosure-policy content itself. Confirmed by reading the spec directly: its only assertion per
  file is `expect(disallowed).toEqual([])` over regex-extracted email tokens. The substantive
  content requirements from `06-03-PLAN.md`'s Task 1 acceptance criteria (private-vulnerability-
  reporting primacy, a 0.x supported-versions table, a coordinated-disclosure window) were checked
  exactly once, via a one-off `<verify><automated>` shell/node command executed AT PLAN TIME — this
  command is not part of `npx nx test github-cache` and never runs again. A future edit that
  silently deleted the "Reporting a Vulnerability" section or the "Supported Versions" table would
  pass every test in the persistent 474-test suite. Generated
  `packages/github-cache/src/governance-docs.spec.ts` (4 tests): re-asserts the private-vulnerability-
  reporting language, the 0.x supported-versions table, a coordinated-disclosure window with a
  concrete day count, and (folding in the GOV-02 LICENSE-content leg, which had the same one-time-
  only exposure) the MIT license text + correct copyright holder in the root `LICENSE`. **Adversarial
  RED proof (test-only mutation, never touching `SECURITY.md`):** temporarily changed the
  supported-versions assertion to require a bogus token (`'BOGUS-VERSION-TOKEN-FOR-RED-PROOF'`) not
  present in the real file and re-ran with `--skip-nx-cache` — failed as expected; restored the
  correct assertion (`toContain('0.x')`) and confirmed green.
- **GOV-02** — CONFIRMED covered for the tarball-inclusion leg (`pack-check.cjs`, verified live at
  06-VERIFICATION.md truth 2: LICENSE present in the 76-file `npm pack --dry-run` list) and now also
  for the content leg via the new `governance-docs.spec.ts` (folded in above, since it scans the
  same root `LICENSE` file the tarball copy mirrors).

**Result:** 7/9 requirements (DOCS-01, DOCS-02, DOCS-03, DOCS-04, DOCS-05, GOV-02, GOV-03) were
already covered by genuinely non-vacuous, adversarially-checkable automated tests, with their
correctly-scoped residual prose/semantic judgments left Manual-Only (not forced into brittle
assertions). 2/9 (DOCS-06, GOV-01) had a real automated-coverage gap on their hardest behavioral
edge — both now filled with new tests, each proven capable of failing via a test-only RED mutation
before being restored to the correct, passing assertion.

tests before this audit; +2 files / +7 tests: `governance-docs.spec.ts` 4 tests +
`consumer-action-runtime.spec.ts` 3 tests). `npx prettier --check` on both new files → clean. No
