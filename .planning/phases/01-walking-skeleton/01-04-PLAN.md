## Phase Goal

**As a** developer or CI job running Nx against a self-hosted remote cache, **I want to** run a
loopback HTTP server that speaks the Nx `GET`/`PUT /v1/cache/{hash}` contract end-to-end against
a trivial in-process store, **so that** the protocol (status codes, auth, hardening) is proven
and conformance-locked before any real storage backend is built.

> Slice for this plan: the lock. A real `serve` process makes the whole thing runnable
> end-to-end (SC4), and the TEST-07 conformance fixture pins the server's behavior to the
> vendored Nx contract (hard `200`, spec-drift guard) so a future `202`-class regression or a
> silent spec change fails the build.

<objective>
Ship the SC4 composition root (`serve.ts`) test-first and lock the protocol with the TEST-07
conformance fixture (D-05). `serve.ts` resolves a port, generates a CSPRNG bearer token, wires
the writable backend into `createCacheServer`, and binds `127.0.0.1` — answering a scripted
GET/PUT round-trip locally (SC4), with the Windows-safe direct-invocation guard and port
validator baked in (Pitfalls 6-8). The conformance fixture vendors the full Nx OpenAPI spec as a
committed file, guards it with a sha256 drift check + a pinned named Nx version (`23.1.0`, never
the spec's permanently-`1.0.0` embedded version — Pitfall 2), and behaviorally asserts the hard
`200` on PUT success plus `401`/`403`/`404`/`409` + `Content-Length` against the real server.

Purpose: TEST-07 (the conformance fixture) and SC4 (a real serve answers a scripted GET/PUT) —
the phase's correctness lock over the contract built in Plans 02-03.

Output: `serve.ts` + `serve.spec.ts`, the vendored spec JSON, `conformance.spec.ts`, and a
minimal `index.ts` barrel, with `npx nx test github-cache` green.
</objective>
