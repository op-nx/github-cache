## Phase Goal

**As a** maintainer of an Nx workspace whose CI runs on GitHub Actions, **I want to** have my
Nx remote cache read and write GitHub's own Actions cache automatically -- writable only on
trusted triggers, read-only everywhere else -- **so that** CI tasks get real cache hits with
nothing extra to host and no untrusted trigger can ever poison the cache.

> Slice for this plan: the supply-chain gate. `@actions/cache` is the first runtime dependency
> this project has ever taken, and its version string is load-bearing (the toolkit
> version-hashes archive paths, so a version bump can silently MISS every restore). Nothing
> real can be built until that dependency is installed under an exact, human-approved pin.

<objective>
Take the first runtime dependency (`@actions/cache` + `@actions/core`) under an EXACT pin, behind
an explicit human legitimacy checkpoint, and lock that pin with a committed spec so a later
`npm install` cannot silently widen it to a range.

Purpose: ROBUST-03(a) -- "`@actions/cache` (and version-hash-sensitive deps) pinned exact (not
`^`)". This is a security control, not a style preference: the package-legitimacy audit returned
**SUS (`too-new`)** for `@actions/cache`, and the toolkit folds the archive path + compression
choice into a version hash whose inputs change across releases.

Output: a non-empty `dependencies` block in `packages/github-cache/package.json`, a synced
lockfile, and `src/pinned-deps.spec.ts` guarding both specifiers.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npm registry -> workspace `node_modules` | Untrusted third-party code enters the build and the CI runtime for the first time in this project. |
| `package.json` specifier -> resolved install | A range specifier lets an unreviewed future release enter without any human decision. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-2-SC | Tampering | npm install of `@actions/cache` / `@actions/core` | high | mitigate | `@actions/cache` carries a **SUS (`too-new`)** legitimacy verdict. Task 1 is a blocking, non-auto-approvable `checkpoint:human-verify` immediately BEFORE the install: the human confirms the canonical `actions/toolkit` provenance, the absence of an install script, and the exact version. Both packages are verified `postinstall`-free. |
| T-2-01 | Tampering | `package.json` dependency specifier | high | mitigate | Both specifiers are installed with `--save-exact` (bare `x.y.z`) and locked by `src/pinned-deps.spec.ts`, which fails the build on any range operator. The end-to-end consequence of an unreviewed bump (a silently MISSing restore) is caught only by the Plan 06 CI canary, so the static pin guard is the first line of defence. |
| T-2-02 | Tampering | `package-lock.json` | medium | mitigate | The lockfile is regenerated by npm, never hand-edited, so integrity hashes are registry-derived. CI installs with `npm ci` (lockfile-exact), so the resolved tree in CI equals the reviewed tree. |
</threat_model>
