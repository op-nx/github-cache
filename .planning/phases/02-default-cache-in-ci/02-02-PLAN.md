## Phase Goal

**As a** maintainer of an Nx workspace whose CI runs on GitHub Actions, **I want to** have my
Nx remote cache read and write GitHub's own Actions cache automatically -- writable only on
trusted triggers, read-only everywhere else -- **so that** CI tasks get real cache hits with
nothing extra to host and no untrusted trigger can ever poison the cache.

> Slice for this plan: the trust boundary itself. After this plan a caller can ask, from a bag
> of environment variables alone, "is this process allowed to write to the cache?" and get a
> default-deny answer that a fork-triggered workflow can never turn into a yes.

<objective>
Build the conservative default-deny write-trust predicate, test-first: `isWriteTrusted(env)` is
true only when the process is running in GitHub Actions AND the triggering event is in the
`TRUSTED_EVENTS` allowlist, which in Phase 2 is exactly `push` and `schedule`.

Purpose: TRUST-03 -- dangerous shared-default-scope events are refused on the write gate,
asserted by test. This is the in-code half of the governing CREEP defence (CVE-2025-36852):
because Phase 2 trusts no fork-reachable trigger at all, the composition is safe by construction
rather than by a server-side backstop.

Output: `src/lib/trust.ts` + `src/lib/trust.spec.ts`, with the allowlist pinned by a
deep-equality assertion so a later widening cannot land silently.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub-provided workflow context -> write authorisation | `GITHUB_EVENT_NAME` / `GITHUB_ACTIONS` are the only inputs that decide whether this process may write to a shared cache. Everything downstream trusts this predicate's answer. |
| fork-controlled trigger -> default-branch cache scope | The class of triggers that run with the base repository's default-scope credentials while executing contributor-influenced context. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-2-03 | Tampering / Elevation | `isWriteTrusted` allowlist (CREEP, CVE-2025-36852) | critical | mitigate | The gate is an allowlist, default-deny, with no denylist path: an unrecognised or unset trigger returns `false`. The allowlist contains only `push` and `schedule`, neither of which is fork-reachable with default-branch cache scope, so Phase 2 is safe by construction rather than relying on a server-side backstop. A deep-equality test pins the allowlist content, so a widening cannot land as an unreviewed one-word edit. |
| T-2-04 | Spoofing | `GITHUB_EVENT_NAME` supplied by the environment | medium | accept | The predicate reads process env, which a compromised runner could set arbitrarily. Accepted for Phase 2: the load-bearing control against a spoofed trigger is GitHub's server-side scope isolation plus the ephemeral single-tenant runner the deployment assumes; the in-code gate is defence-in-depth. This only becomes decision-relevant when Phase 5 widens to contributor-facing triggers. |
| T-2-05 | Tampering | allowlist duplication drift | medium | mitigate | Exactly one `TRUSTED_EVENTS` declaration exists (asserted by a repository-wide count in the acceptance criteria). A second, dependency-free copy for a pre-install action context is deliberately deferred to Phase 5 together with its parity assertion, rather than introduced here without one. |
| T-2-SC | Tampering | npm/pip/cargo installs | high | accept | No package-manager install occurs in this plan; the only dependency delta in Phase 2 was installed under a blocking human legitimacy checkpoint in Plan 02-01. Accepted -- the install vector does not occur here. |
</threat_model>
