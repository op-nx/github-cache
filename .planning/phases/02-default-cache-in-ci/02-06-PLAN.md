## Phase Goal

**As a** maintainer of an Nx workspace whose CI runs on GitHub Actions, **I want to** have my
Nx remote cache read and write GitHub's own Actions cache automatically -- writable only on
trusted triggers, read-only everywhere else -- **so that** CI tasks get real cache hits with
nothing extra to host and no untrusted trigger can ever poison the cache.

> Slice for this plan: the proof. Every earlier plan in this phase is verified against a mocked
> toolkit. This plan makes the capability real for the first time: a push to the default branch
> writes to GitHub's cache service and a second job reads it back.

<objective>
Dogfood the whole phase in this repository's own CI: a minimal JS action runs `serve()` in its
own foreground process and drives a scripted PUT in a seed job, then a scripted GET in a
dependent verify job keyed on the same workflow run id, asserting a real cache hit.

Purpose: ROADMAP SC5 and ROBUST-03(b). This is simultaneously the phase's headline capability
proof AND the upgrade canary for `@actions/cache` -- a version bump that changes the archive
version hash shows up here as a failed verify job instead of as a silently degraded cache.

Output: `packages/github-cache/action.yml`, `src/action/index.ts`, a `test:act` script, a
`.fallowrc.jsonc` entry, and two new jobs in `.github/workflows/ci.yml`.
</objective>

## Artifacts this phase produces

See `02-01-PLAN.md` "## Artifacts this phase produces" for the full Phase 2 symbol list. This
plan creates `packages/github-cache/action.yml` (inputs `hash`, `operation`),
`packages/github-cache/src/action/index.ts`, the root `test:act` npm script, and the
`dogfood-seed` / `dogfood-verify` CI jobs.

**Resolved planning decision (do not re-open): the hit is asserted with a two-job seed then
verify pair keyed on the workflow run id, not with a single-job save then restore.** A same-run,
same-process restore can succeed from local state and would let a completely broken upload look
green -- and the research pass explicitly flags same-run restore reliability as an assumption
rather than a verified fact. Two jobs make the assertion deterministic and prove the round-trip
actually crossed GitHub's cache service, which is what "real hits and misses" means. The cost is
one extra job.

**Resolved planning decision (do not re-open): the action runs `serve()` in its own foreground
process.** No detached child process, and no use of the newer background-step keywords. A
foreground run sidesteps the Windows detached-stdio termination trap entirely, and the
background-step consumption pattern is deliberately Phase 6 work. The action is INTERNAL to this
repository's dogfood; the published, enumerated public surface is Phase 6.

**Resolved planning decision (do not re-open): the `test:act` script keeps its name and becomes a
self-skipping wrapper.** Running the cache canary locally is infeasible -- a local runner's cache
server speaks only the superseded protocol, and this development host would emulate the runner
image under a foreign architecture besides. So `test:act` invokes the same built action entry the
CI job uses, and that entry exits successfully with a printed skip notice when the Actions cache
runtime variables are absent. The real gate is the CI job.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| action runtime -> `serve` process | The only place the runtime cache credentials exist. They must reach the server by inheritance and go no further. |
| workflow log -> reader | Anything printed by the server or the action is world-readable on a public repository. |
| CI job -> GitHub's shared cache | The seed job performs the first real write this project has ever made to shared, repository-scoped storage. |
| workflow trigger set -> which jobs run | The trigger deciding whether these jobs run at all, and therefore whether a write is attempted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-2-18 | Information disclosure | runtime cache credentials re-exported through the workflow environment file | high | mitigate | The credentials are consumed only from the inherited process environment. The common re-export workaround is forbidden and its absence is asserted by a zero-count check on both the action entry and the workflow file. This constraint is the whole reason an action runtime, rather than an ordinary shell step, is the launch path. |
| T-2-19 | Information disclosure | bearer token printed to a public workflow log | high | mitigate | The action masks the bearer token through the toolkit's secret-masking call as its first action after the server starts, before any code path can write it out. The existing direct-invocation path that prints the token is not used by the action. |
| T-2-20 | Tampering | a silently passing dogfood job | high | mitigate | Every branch of the action either asserts an exact expected status and body or fails the job explicitly; a miss produces a named failure rather than a skip. The two-job split means a broken upload cannot be masked by a local read-back. Without this, the phase's headline capability could be reported green while caching nothing -- the precise failure mode this requirement exists to prevent. |
| T-2-21 | Elevation of privilege | the dogfood job attempting a write on an untrusted trigger | high | mitigate | The jobs are restricted to the default-branch push trigger, and independently the selection function would hand back a read-only backend on any other trigger, so a write is refused twice over. No new workflow trigger is added and the write allowlist is not widened; widening remains Phase 5 work. |
| T-2-22 | Denial of service | a job-level permission block silently dropping a scope | medium | mitigate | No job-level permission block is added, asserted by a count check, so the workflow-level grant is not replaced. The hit assertion reads the response of the local server rather than querying a management endpoint, so no additional scope is required at all. |
| T-2-23 | Tampering | an unreviewed cache dependency upgrade changing the archive version derivation | high | mitigate | This job pair is the end-to-end gate the pin requirement calls for: an upgrade that changes the archive version derivation surfaces here as a failing verify job. The static pin guard from Plan 02-01 prevents an unattended upgrade in the first place; this job catches a deliberate one that breaks restores. |
| T-2-SC | Tampering | npm/pip/cargo installs | high | accept | No new package-manager dependency is added in this plan; the CI jobs install from the committed lockfile. The only dependency delta in this phase was installed in Plan 02-01 under a blocking human legitimacy checkpoint. Accepted -- the install vector does not occur here. |
</threat_model>
