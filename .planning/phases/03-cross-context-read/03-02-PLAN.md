## Phase Goal

**As a** developer working on any OS, **I want to** read this repo's CI-produced cache locally
through the GitHub Releases reader using my existing GitHub auth, **so that** I get cache hits
without ever being served a wrong-OS artifact.

<objective>
Deliver the credential half of the read slice: the three-tier local auth chain and the local repo
identity, both resolved from the developer's existing GitHub setup and both degrading to
`undefined` -- never to an anonymous request -- when nothing resolves.

This is the FOUND-02 core. It is the only new ground in the phase with no in-repo precedent: the
package currently spawns no subprocesses at all.

Purpose: a private-repo read must work from the developer's existing auth, and must never
silently drop to the anonymous 60/hr tier that cannot see a private repo at all.

Output: `local-context.ts` (+spec), plus one `export` keyword added in `select-backend.ts`.
</objective>

    helper would otherwise wedge the developer's build indefinitely, and the safe direction on a

    Implement ONE module-private helper,
    `runHelper(file: string, args: readonly string[], stdin?: string): Promise<string | undefined>`,
    built on `spawn` from `node:child_process`. Use `spawn` for all three call sites rather than
    mixing `execFile` for two of them and `spawn` for the one that needs stdin: `git credential
    fill` requires stdin regardless, and one mechanism means one hardening surface, one mock shape,
    and no dependence on the `promisify` custom-symbol behaviour that a mocked module does not
    carry. This is a deliberate consolidation of the two-mechanism sketch in RESEARCH Pattern 1;
    the four load-bearing options are identical either way. Mark it with a `ponytail:` comment.

    The four options are non-negotiable and each needs its own inline reason:
    `shell` false (injection-safe; an explicit argv array is passed, never an interpolated command
    string, and a native binary resolves from PATH with no quoting even when its directory contains
    spaces); `timeout` set to `HELPER_TIMEOUT_MS`; `windowsHide` true; and an `env` spreading
    `process.env` then setting `GIT_TERMINAL_PROMPT` to `0` and blanking both `GIT_ASKPASS` and
    `SSH_ASKPASS`. All three env keys are load-bearing together -- disabling terminal prompts alone
    still let git reach for its askpass executable in the verified probe, which can pop a modal
    dialog on a desktop and block the build.

    `runHelper` accumulates stdout only. It deliberately does not attach a `stderr` listener at
    all. Give that omission a comment stating the measured reason: helper failure stderr is
    LOCALIZED to the system language -- it came back in Danish on the probe machine -- so any
    stderr sentinel silently misfires for every non-English developer, and stderr can additionally
    carry credential-adjacent material. Discrimination is structural only.

    Resolve `undefined` on the child `error` event (the missing-binary shape) and on any close code
    other than 0. On close code 0, resolve the trimmed stdout coalesced with `|| undefined` so an
    empty result falls through exactly as the repo's other credential chains do. Never inspect an
    error code property to decide -- it is a number for a non-zero exit and the string `ENOENT`
    when the binary is absent, so it is overloaded and every failure means the same thing to the
    caller: this tier yielded nothing, try the next. Write `stdin` to the child and end it when the
    caller supplied one.

    Export `resolveLocalReadToken(env: NodeJS.ProcessEnv = process.env): Promise<string | undefined>`
    implementing D-08's three tiers in order, each with an early return and a blank line around it.
    Tier 1 calls `resolveGitHubToken(env)` imported from `./select-backend.js` -- unchanged, not
    re-implemented and not extended in place, because it is env-only by design, shared with the CI
    write path, and its fallthrough is pinned by TEST-01. Tier 2 is `runHelper` on the gh CLI with
    the auth-token argv. Tier 3 is `runHelper` on git with the credential-fill argv, supplying the
    key-value request terminated by a blank line, then extracting the password field structurally
    with an anchored multiline regex over the returned key-value lines and coalescing with
    `|| undefined`. Comment tier 3's position: it is last because it is the slowest and the only
    one that can touch a keychain.

    Close the function with an explicit comment that reaching the end means every tier is exhausted
    and the answer is `undefined`, and that there is deliberately no anonymous fallback (FOUND-02,
    D-09) -- the reader misses instead, because an unauthenticated request cannot see a private
    repo anyway and would silently bind this package to a 60/hr tier.

    Never log, echo, or interpolate a resolved token anywhere.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| process env -> credential chain | Environment values are attacker-influenceable in a compromised shell profile or a malicious `.env` loader. |
| Node -> spawned `gh` / `git` | Arguments cross into another process; a shell would make them a command line. |
| git remote URL -> repo identity | The remote URL is repository-controlled content that becomes a URL path segment in Plan 03. |
| credential helper -> resolved token | A secret crosses into this process's memory. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-07 | Tampering / Elevation of Privilege | `runHelper` spawn | high | mitigate | `shell` false with an explicit argv array at every call site; no value is ever interpolated into a command string, so there is no shell metacharacter surface. Asserted by a source gate (`shell: true` count is 0). |
| T-03-08 | Information Disclosure | resolved token | high | mitigate | The token is returned to the caller and never logged, echoed, interpolated into a message, or written to any stream. `stderr` is not even captured. |
| T-03-09 | Information Disclosure | helper stderr | medium | mitigate | No `stderr` listener is attached. Helper stderr is localized and can carry credential-adjacent material; discrimination is structural (close code plus non-empty stdout) only. |
| T-03-10 | Denial of Service | `git credential fill` | high | mitigate | Bounded `timeout`, `GIT_TERMINAL_PROMPT` disabled, and both askpass variables neutralised -- all three together, since prompt-disabling alone still let git reach for its askpass executable in the verified probe. |
| T-03-11 | Spoofing | repo identity | high | mitigate | The env override must satisfy `GITHUB_REPOSITORY_PATTERN`; the remote URL must match an anchored GitHub owner/name regex. An unparseable identity yields `undefined` and a MISS -- never a guess that would resolve into another repository's cache namespace. |
| T-03-12 | Information Disclosure | anonymous downgrade | high | mitigate | D-09: no unauthenticated fallback exists in the chain. An exhausted chain returns `undefined`, asserted by a dedicated test. |
| T-03-13 | Repudiation | helper failure attribution | low | accept | A failed tier is indistinguishable in the return value from an absent tool by design; every failure means the same thing to the caller. Fine-grained attribution would require reading localized stderr, which is the larger hazard. |
</threat_model>
