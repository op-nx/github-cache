## Phase Goal

**As a** developer working on any OS, **I want to** read this repo's CI-produced cache locally
through the GitHub Releases reader using my existing GitHub auth, **so that** I get cache hits
without ever being served a wrong-OS artifact.

<objective>
Close the slice end to end: build the real default `ReleaseReadClient` (the authenticated GitHub
REST sequence) and wire the reader into `selectBackend`'s local branch, so a developer actually
reads a cache entry back -- the capability the two Wave 1 plans build the parts for.

This is the phase's integration and its highest-risk detail: `selectBackend` is synchronous and
its zero-arity is structurally pinned, while token and repo resolution are async. The async work
therefore defers into the client's `fetchAsset`, not into construction.

Purpose: D-01 (wire the reader) and D-03 (the real fetch client) are what make FOUND-02 and
CORR-01 observable to a real developer rather than only to a fake in a unit test.

Output: the real default client added to `releases-backend.ts` (+spec), the three-line
`select-backend.ts` edit (+spec extension).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> api.github.com | The resolved token is attached to outbound HTTPS requests; a redirect could carry it to a third-party origin. |
| asset name + repo -> URL path | An attacker-influenced asset name or remote-derived repo becomes a URL path segment. |
| GitHub response -> Buffer | A hostile or oversized response body is read into memory. |
| selectBackend construction | The sync/async boundary: async credential work must not leak into the synchronous, zero-arity contract. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-14 | Information Disclosure | asset download redirect | high | mitigate | Rely on fetch's spec-mandated cross-origin `Authorization` drop; forbid `redirect: 'manual'` and header re-attachment (source gate asserts the option is absent). The signed redirect target carries its own auth. |
| T-03-15 | Information Disclosure | anonymous downgrade | high | mitigate | D-09: `fetchAsset` returns undefined before any request when no token resolves; asserted by a zero-fetch-calls test. A private repo is never probed unauthenticated. |
| T-03-16 | Denial of Service | REST fault handling | high | mitigate | Every non-404 fault throws and the port degrades it to a MISS; a 404 is a silent MISS; no retry or backoff, so a rate-limited or offline GitHub can never break or stall the build (SRV-05, D-11). |
| T-03-17 | Tampering | repo identity in URL path | medium | mitigate | The repo segment comes only from `resolveRepoIdentity` (Plan 02: validated env override or an anchored GitHub owner/name regex), and the hash is server-validated `^[a-f0-9]{1,512}$` before reaching the backend, so both URL segments are bounded. |
| T-03-18 | Information Disclosure | oversized asset body | low | accept | The response is buffered; the existing 2 GB server body cap (SRV-04, Phase 1) bounds what the server will forward, and Phase 3 writes nothing to disk. A dedicated read-side size cap is a Phase 4 concern. |
| T-03-19 | Elevation of Privilege | selectBackend contract drift | high | mitigate | The edit adds no parameter and does not make the function async; `Function.length === 0` and the synchronous return are both asserted, so the no-flag safety property (TRUST-05) and the `serve.ts` call site are preserved. |
</threat_model>
