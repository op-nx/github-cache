## Phase Goal

**As a** repo maintainer running Nx builds in CI, **I want to** mirror CI-produced cache entries to GitHub Releases on default-branch pushes and prune them safely by age, with loud failures and graceful cap handling, **so that** cross-context readers get correct cross-OS hits and the cache never breaks a build or deletes live data.

This is the headline slice: on a default-branch push, CI actually mirrors entries to Releases per-OS, and a live read-back proves the round-trip.

## Wiring caveat (load-bearing, flag before implementing)

RESEARCH's structure note says "runs the built dist/publish/index.js" (a `run:` step). The repo's own Phase-2 empirical finding (packages/github-cache/src/action/index.ts:28 comment) contradicts that: `ACTIONS_RUNTIME_TOKEN`/`ACTIONS_RESULTS_URL` are injected ONLY into a JS-action runtime, so `@actions/cache` `restoreCache` silently no-ops/MISSes in a plain `run:` step. The publish path RESTORES cache bytes, so it MUST run in a node24 JS-action context.

**Recommended mechanism:** add an `operation: publish` value to the existing `packages/github-cache/action.yml` (main already points at `dist/action/index.js`, node24) and a publish branch in `action/index.ts`. This reuses the proven runtime, needs no `../` in the action `main` path (a dedicated sibling action.yml cannot traverse out to `dist/`, and only one action.yml per directory), and fits the action's "internal CI" scope. If the executor confirms a dedicated publish action is preferred, it must resolve the build-output/`main` path first. Do NOT silently choose a `run:` step.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| workflow trigger -> live publish | Only a trusted {push,schedule}+default-branch run may mirror to public Release assets (runtime enforcement of the sync gate) |
| job token grant -> Actions-cache + Releases | Least-privilege, both-scopes-restated credential; a dropped scope is a silent failure, an over-scope is elevation |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-18 | Elevation of Privilege | untrusted trigger reaching publish | high | mitigate | isSyncTrusted enforced as the FIRST statement of the publish branch (D-01) + the workflow `if: push` as defense-in-depth; gated-out = exit 0, not an error (TRUST-02, C2) |
| T-04-19 | Denial of Service | dropped actions:read scope | medium | mitigate | Job-level permissions RESTATE both contents: write AND actions: read (a job block replaces the workflow grant wholesale) — Pitfall 3 |
| T-04-20 | Tampering | collapsed per-OS matrix / test-coupled publish | medium | mitigate | fail-fast:false [ubuntu,windows] matrix, each leg mirrors only its own OS (D-03); needs: build not test so a test failure never drops built entries |
| T-04-21 | Information Disclosure | token in action logs | low | mitigate | Token by process inheritance (GITHUB_TOKEN env), resolveGitHubToken; never interpolated into summaries/annotations (existing discipline) |
</threat_model>
