## Phase Goal

**As a** repository maintainer adopting @op-nx/github-cache, **I want to** have write-trust widened to `pull_request`/`release` only where GitHub's default-branch cache guard exists (fail-closed on GHES), backed by a single-source allowlist, a server-produced-key mirror filter, and an installable PPE-hygiene gate, **so that** contributor PR/release runs can write cache without opening a CREEP hole and only genuine cache keys are ever mirrored.

<objective>
Widen the Phase 2 write gate (`isWriteTrusted`/`TRUSTED_EVENTS` in `src/lib/trust.ts`) to admit `pull_request` and `release` - but ONLY on hosts where GitHub's 2026-06-26 server-side read-only-default-branch-cache guard exists, inferred structurally from `GITHUB_SERVER_URL` (host `github.com` or a real `*.ghe.com` subdomain), failing closed on every GHES host and on any malformed/missing value. Prove the widening end-to-end through `selectBackend` and prove it did NOT leak into the separate sync gate.

Purpose: TRUST-01 / ADR C1 - contributor PR/release runs can write cache where GitHub's guard makes it safe, with GHES fail-closed. The in-code host gate is fork-spoofable defense-in-depth; the load-bearing control is GitHub's server-side read-only-token guard + scope isolation.
Output: widened trust.ts (single source, D-05); extended trust.spec.ts host matrix; corrected select-backend.spec.ts integration under widening.
</objective>
