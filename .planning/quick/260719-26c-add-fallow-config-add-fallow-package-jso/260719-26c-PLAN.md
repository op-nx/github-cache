<objective>
Make Fallow a first-class, CI-gated, project-owned tool: a committed `.fallowrc.jsonc`
that declares the known import-graph false-positives so a whole-repo dead-code analysis
is clean, a pinned `fallow` devDependency + npm scripts, and a `fallow` CI job that gates
every push/PR.

Purpose: Fallow's structural (dead-code / reachability) pass already caught two real
Phase 1 bugs (CR-01 crash, WR-03 auth bypass). This wires that analysis into CI so the
whole team gets it on every change, project-owned rather than depending on a global install.

Output: `.fallowrc.jsonc`, `fallow` devDep + scripts in package.json (+ lockfile), a
`fallow` job in ci.yml, and a `.gitignore` entry for the cache.

DECISION — CI gate command is `fallow dead-code --fail-on-issues`, NOT `fallow audit --ci`
(the task text's suggestion). This was verified empirically against the installed
fallow 3.6.0 on this repo, and `fallow audit` is the wrong tool for a whole-repo CI gate here:
  1. `fallow audit` FAILS OPEN: with a bogus/unresolvable base ref it prints an error and
     exits 0 (verified). A default `actions/checkout@v7` PR checkout is shallow and has no
     `origin/main` ref, so `fallow audit` would silently pass without auditing anything.
  2. `fallow audit` gates NOTHING on push-to-main (merge-base == HEAD -> empty changed set).
  3. `fallow audit` is changed-files + complexity + duplication + styling. On this repo it
     currently FAILS on a pre-existing complexity finding (server.ts `handlePut`, crap 37.1 >
     default 30) that is OUT OF SCOPE for a config/CI task (would require refactoring Phase 1
     code or masking a real signal by raising a threshold).
  4. `fallow dead-code --fail-on-issues` is whole-repo, needs no git base, never fails open,
     gates on push AND pr, and covers exactly the enumerated false-positive classes (unused
     files, unused deps, unused exports) plus the reachability bug class that motivated adoption.
`fallow audit` remains available ad-hoc for local PR-diff review (`npx fallow audit`); a
future task can widen the gate to complexity/duplication once those are tuned/refactored.
This deviation honors the task's stated GOAL ("gate the whole repo, config-declared-clean,
rather than --changed-since; PR checkouts lack a local main ref") over its literal command.
</objective>
